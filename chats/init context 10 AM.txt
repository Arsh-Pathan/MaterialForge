  MaterialForge - Full Codebase Summary

  Project: A reinforcement learning environment for autonomous crystalline
  structure discovery, built for the Meta PyTorch OpenEnv Hackathon. Agents
  place atoms (Metal/A, Conductor/B, Ceramic/C, Polymer/P) on an 8x8 lattice to
  match target physical properties (Hardness, Conductivity, Thermal Resistance,
  Elasticity).

  Architecture

  MaterialForge/
  ├── environment/          # Core physics engine
  │   ├── config.py         # Constants: grid size (8), atom types, costs,
  difficulty presets
  │   ├── lattice.py        # Lattice class: 8x8 grid with place/replace/remove
  operations
  │   ├── physics.py        # Property estimation, phase classification,
  stability/order scoring
  │   └── rubrics.py        # HeuristicRewardRubric: 50% property match, 25%
  stability, 15% order, 10% phase
  ├── server/               # FastAPI server (OpenEnv-compliant)
  │   ├── app.py            # Entry point: creates FastAPI app, mounts static
  files, /playground route
  │   ├── material_forge_env_environment.py  # MaterialForgeEnvironment:
  reset/step logic, done detection
  │   └── static/           # "Discovery Lab" dashboard (HTML/CSS/JS with
  Chart.js)
  ├── scenarios/
  │   └── scenarios.py      # 5 named scenarios (diamond-like, conductor,
  heat-shield, flexible-polymer, balanced-alloy)
  ├── models.py             # Pydantic models: MaterialForgeAction,
  MaterialForgeObservation
  ├── client.py             # WebSocket client (EnvClient subclass)
  ├── inference.py          # Agent: heuristic lookahead + LLM
  (OpenAI-compatible) for action selection
  ├── tests/
  │   └── benchmark.py      # Analytical benchmark suite (100-trial evaluations,
   Ollama support)
  ├── openenv.yaml          # 3 tasks: basic-synthesis, diamond-like,
  superconductor-analogue
  ├── Dockerfile            # Multi-stage Docker build for HF Spaces deployment
  └── pyproject.toml        # Python 3.10+, depends on openenv-core, openai,
  gradio

  Key Components

  - Physics engine (physics.py): Calculates properties from atom fractions +
  neighbor bonding + BFS percolation (for conductivity). Classifies phase via
  2x2 pattern repetition analysis. Computes structural stability (coordination
  energy + mirror symmetry) and lattice order (Bragg-like + quadrant
  homogeneity).
  - Inference agent (inference.py): Generates candidate actions via heuristic
  scoring (clustering, center placement), simulates each locally via the physics
   engine, then asks an LLM to pick the best from top-8 candidates. Falls back
  to pure heuristic if LLM fails.
  - Frontend dashboard: Dark-themed professional UI with real-time lattice
  canvas, property alignment bars, reward convergence chart, KPIs, and analytics
   tab. WebSocket-connected to the server.
  - 3 difficulty levels: Easy (tolerance 20, budget 120), Medium (tolerance 10,
  budget 80), Hard (tolerance 5, budget 60).

  SECTION 3 — Training Requirements

  Is RL training expected?

  YES — absolutely mandatory. The documents are explicit:

  ▎ "The bar isn't 'training script exists.' The bar is 'training script runs

  Training evidence is worth 20% of the score directly, and the training
  pipeline is another 10%.

  Recommended stack:

  - TRL (GRPOTrainer) — the primary training framework
  - Unsloth — for memory efficiency (QLoRA/LoRA), especially on limited hardware
  - GRPO algorithm — preferred over PPO for LLM RL; more memory-efficient, no
  value model needed
  - OpenEnv environment_factory pattern — wraps your environment as tool methods
   the trainer discovers automatically

  How training should work (from TRL docs):

  1. Create an environment class with reset() and tool methods (e.g.,
  place_atom(), remove_atom())
  2. The GRPOTrainer discovers tool methods, generates completions, parses tool
  calls, executes them, feeds results back
  3. Reward comes from the environment after each episode
  4. Model: Start with a capable instruct model (e.g., Qwen/Qwen3-0.6B or
  Qwen/Qwen2.5-3B)

  Expected evaluation metrics:

  - Reward curves over training steps (must be plotted, labeled, saved as
  images)
  - Before/after behavior comparison (untrained baseline vs. trained agent)
  - Loss plots
  - Quantitative comparison: random agent vs. heuristic baseline vs. trained LLM
   agent

  Key warnings from organizers:

  - Don't naively upcast 4-bit to 16-bit when saving LoRA — use proper merge
  path
  - Start with easy tasks, use curriculum learning
  - Binary rewards (1.0 success / 0.0 fail) often work better than shaped
  rewards for GRPO
  - max_completion_length must be increased for long episodes (default is too
  small)
  - Test reward function before training — if a capable model can't beat random,
